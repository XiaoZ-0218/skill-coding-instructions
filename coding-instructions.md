# 编程与 Git 工作流规范

## 一、Andrej Karpathy 编程准则（硬性规则）

写代码时必须遵循以下 Andrej Karpathy 编程哲学。此规则仅适用于写代码，不适用于对话交流。

### 1. 极简主义 — 从简开始
- 每一段代码必须尽可能短、自包含、零依赖（或接近零依赖）。
- 先写最简版本（"原子的"实现），再逐步优化。
- 绝不无缘无故增加复杂度。

### 2. "不要做英雄" — 先复制，再创新
- 不要在一开始发明新奇方案。先找最相关的成熟方案，复制粘贴。
- "一次只增加一个复杂度" — 不要一开始就把所有东西都堆上去。

### 3. 理解底层原理
- 不要盲目依赖抽象层。必须理解底层原理，否则失败时无法调试。
- 代码要体现对原理的理解，而不是黑盒调用。

### 4. 从简单到复杂 — 循序渐进
- 先搭建最简单的可行方案，验证通过后再逐步增加复杂度。
- 先让代码能跑通，再考虑优化。

### 5. 简洁代码风格
- 保持文件短小精悍，每个文件职责单一。
- 扁平文件结构，最小化目录嵌套，减少不必要的抽象层级。
- 包含丰富且有教育意义的注释。

### 6. 最小依赖
- 优先使用标准库，减少外部依赖。
- 纯实现优先于框架调用。

## 二、Git 提交与分支工作流规范

### 前置规则 — 新建项目初始化

#### 0. 初始化项目仓库
- **新建项目时，必须先初始化 Git 仓库**，再开始编写任何代码。
- 执行 `git init` 初始化本地仓库。
- 创建 `.gitignore` 文件（根据项目类型添加合适的忽略规则）。
- 创建 `README.md` 文件，包含项目名称和简要描述。
- 执行 `git add -A && git commit -m "🎉 init: 初始化项目"` 完成首次提交。
- **不要自动关联远程仓库**，用户后续会手动操作。

### 分支工作流 — 新功能开发与合并

#### 1. 新功能必须基于功能分支开发
- **开始新功能前，从主分支（`main` / `master`）切出新分支**，不要在主分支上直接开发新功能。
- 功能分支命名规范：
  - `feature/<简短描述>`：新功能开发，例如 `feature/user-login`
  - `fix/<简短描述>`：Bug 修复，例如 `fix/null-pointer-in-auth`
  - `test/<简短描述>`：测试相关改动，例如 `test/add-unit-tests`
  - `docs/<简短描述>`：文档更新，例如 `docs/update-readme`

#### 2. 功能分支上的提交规范
- 功能分支上同样遵循本规范的 Gitmoji 提交格式。
- **每完成一个逻辑步骤就提交一次**，保持提交历史清晰。
- 如果任务中途暂停，使用 `🚧 wip` 提交保存进度。

#### 3. 合并前必须完成全部测试
- **功能开发完成后，必须先运行项目的完整测试套件**（单元测试、集成测试、端到端测试等，视项目而定）。
- 所有测试通过、代码无明显问题后，**必须向用户汇报测试结果并明确请求合并许可**。
- **只有在用户明确同意后，才允许执行合并操作**。禁止在未获用户许可的情况下自行合并回主分支。
- 如项目配置了 CI/CD 流水线，必须等待流水线全部通过。

#### 4. 合并回主分支
- **合并前，必须先向用户请求合并许可**，汇报功能分支的改动摘要和测试结果，等待用户明确同意后才能执行合并。
- 获得用户许可后，先用 `git merge main` 将主分支最新代码同步到功能分支，处理可能的冲突。
  - **禁止使用 `git rebase` 将功能分支合并回主分支**。合并回主分支一律使用 **Merge commit（合并提交）**，保留完整合并记录与合并节点。
- 合并方式：
  - **Merge commit（合并提交）**：会生成一个合并节点，保留完整的分支开发历史。这是唯一允许的合并方式。
  - **禁止使用 Squash and merge（压缩合并）和 Rebase and merge（变基合并）**，这两种方式不会生成合并节点，会丢失分支历史。
- 合并提交的标题建议：`<emoji> <type>: <中文功能描述>`，例如 `✨ feat: 完成用户登录功能`。

#### 4.1 rebase 的使用场景
- **`git rebase` 仅用于以下场景**：当多个并行的功能分支先后合并回主分支后，尚未合并的支线分支需要将主分支最新变更纳入自己分支时，使用 `git rebase main` 保持提交历史线性整洁。
- **严禁在任何情况下用 rebase 替代 merge 来合并分支到主分支**。

#### 5. 合并后清理
- 合并完成后，**删除已合并的功能分支**（本地与远程），保持仓库分支整洁。
- 回到主分支继续后续开发。

### 核心原则

- **频繁提交**：每完成一个逻辑步骤就提交一次，包括但不限于：
  - ✅ 新功能开发完成
  - ✅ Bug 修复
  - ✅ 重构完成
  - ✅ WIP 草稿（工作中保存进度）
  - ✅ 依赖更新
  - ✅ 文档/注释更新
  - ✅ 测试通过
  - ✅ 样式/UI 调整
- **宁多勿少**：不要攒多个改动一起提交，每个提交只做一件事
- **提交信息格式**：`<emoji> <type>: <描述>`

### Gitmoji 速查表

| Emoji | Type | 使用场景 |
|-------|------|----------|
| ✨ | feat | 新功能、新特性 |
| 🐛 | fix | Bug 修复 |
| ♻️ | refactor | 代码重构（不涉及功能变化） |
| 💄 | style | UI/样式调整 |
| 📝 | docs | 文档、注释更新 |
| 🚧 | wip | 工作进行中，暂存进度 |
| ⚡ | perf | 性能优化 |
| ✅ | test | 添加或更新测试 |
| 📦 | deps | 依赖安装/更新/移除 |
| 🔧 | config | 配置文件修改 |
| 🚀 | deploy | 部署相关 |
| 🎨 | format | 代码格式化 |
| 🏗️ | build | 构建系统/脚本修改 |
| 🔥 | remove | 删除代码/文件 |
| 🚚 | move | 移动/重命名文件 |
| ⏪ | revert | 回退更改 |

### 示例

```
✨ feat: 添加用户登录功能
🐛 fix: 修复密码重置接口空指针异常
♻️ refactor: 提取用户认证逻辑到独立服务
💄 style: 调整首页卡片阴影效果
📝 docs: 更新 API 接口文档
🚧 wip: 购物车模块开发中
✅ test: 添加用户注册单元测试
📦 deps: 安装 axios 依赖
```

### AI 代理行为守则

1. **新功能/修复开始前，先从主分支切出功能分支**，遵循 `feature/<描述>`、`fix/<描述>` 等命名规范。
2. **每次代码变更后主动提交**，不要等待用户要求才提交。
3. 提交信息要**简洁但有意义**，能看出改了什么东西。
4. 如果用户要求"保存进度"或"先放着"，使用 `🚧 wip` 提交。
5. 多个修改意图混在一起时，**拆分成多个提交**。
6. **合并回主分支前，必须先运行完整测试并确保通过**；测试未通过时不得合并。
7. **测试通过后，必须向用户汇报测试结果和改动摘要，明确请求合并许可**。只有在用户明确同意后，才能执行合并操作。禁止自行合并。
8. 功能分支合并并清理完成后，回到主分支继续后续工作。
